# ADR-0007: README 生成粒度

> 应用自动维护"分类目录级"和"仓库根级"两层 README；单文件级别用用户可选的伴生 `<name>.md` 备注。
>
> 状态：Accepted
> 日期：2026-04-26
> 影响范围：core/readme / core/note
> 关联 ADR：—

## 上下文

用户原始诉求："每个文件都有一个 readme.md，可以清晰可见更改改动。"

直译就是"每个文件一个 readme.md"，但实际操作中有歧义：

- **每文件 README** = 每导入一个文件就在旁边生成一个 `<name>.md`？还是 `<name>.readme.md`？
- **目录级 README** = 每个分类目录下一个 README.md，列出该目录文件？
- **仓库级 README** = 仓库根有一个总览 README.md？

不同粒度各有取舍。需要在"用户体验直观度"、"维护成本"、"FS 体面度"之间做选择。

## 决定

采用 **三层结构**：

| 层级 | 文件 | 是否自动 | 用户可编辑区域 |
|---|---|---|---|
| 仓库根 | `~/AreaMatrix/README.md` | 自动 | 标记块外的一切 |
| 分类目录 | `~/AreaMatrix/docs/README.md` | 自动 | 标记块外的一切 |
| 单文件备注 | `<name>.md`（与文件同目录、同名） | 用户主动创建 | 全部 |

**自动生成区域**用 `<!-- AREAMATRIX:BEGIN -->` 与 `<!-- AREAMATRIX:END -->` 标记，应用只覆盖标记块内的内容。

**单文件备注**完全由用户手动写：

- UI 中"详情面板"里编辑 → 应用写到 FS 上的 `<name>.md`
- 用户也可在 Finder 直接编辑 `<name>.md`
- 应用不擅自创建 `<name>.md`，只在用户首次输入备注时创建

**改动历史**记录在 DB 的 `change_log` 表，UI 中"改动历史"标签展示。**不**写到 README 里（避免高频改动 = 高频文件写入 = FSEvents 风暴）。

## 理由

1. **平衡用户期望与可行性**：目录级 README 直接可见 + 用户备注按需，组合等价于"每文件可读"
2. **避免文件爆炸**：每文件强制生成 `.md` 会让目录文件数翻倍，用户在 Finder 看着杂乱
3. **保护用户内容**：标记块机制让应用与用户内容互不干扰
4. **改动历史在 DB 不在 FS**：高频改动写文件会触发 FSEvents 风暴，且 git diff 噪声大
5. **用户备注与改动历史分离**：用户写"为什么留这个文件"是主观备注；"什么时候被改了"是客观历史，二者性质不同
6. **README.md 是 GitHub / VSCode 等的天然入口**：方便用户在外部工具浏览仓库

## 考虑过的备选

### A. 每个文件强制生成 `<name>.md`

- 优点：直接对应用户原始诉求
- 缺点：
  - 文件数翻倍，Finder 视图杂乱
  - 短文件名场景下歧义（`README.md` 旁边再来 `README.md.md`）
  - 高频改动写入压力大
- **为什么没选**：体验差且实现复杂

### B. 仓库级 README + 单文件 `<name>.md` 双层

无目录级 README。

- 优点：层级少
- 缺点：
  - Finder 进入分类目录时无导览
  - 仓库 README 列全部文件会很长
- **为什么没选**：分类目录级 README 是"中间地图"，价值大

### C. 单文件 changelog 写到 `<name>.md`

每文件备注 + 改动历史合并到一个 `.md`，应用维护历史段。

- 优点：用户单文件视图最完整
- 缺点：
  - 高频改动 → 高频文件写入 → FSEvents 风暴 + git diff 噪声
  - 应用与用户内容混在一个文件，标记块会很复杂
- **为什么没选**：性能与可维护性都差

### D. 不生成任何 README，全部 UI 内显示

完全不写 README 文件。

- 优点：FS 干净
- 缺点：
  - 不打开应用就看不到任何上下文（违反"本地优先 + 用户拥有数据"）
  - 不利于 git / GitHub / VSCode 等外部浏览
- **为什么没选**：违反产品哲学

### E. 用扩展属性（xattr）存元数据

把分类、备注存到 macOS 扩展属性里，不生成 `.md`。

- 优点：FS 看着最干净
- 缺点：
  - 跨工具复制 / iCloud 同步可能丢失
  - 不可见 = 用户不知道有数据
  - 不便于版本控制
- **为什么没选**：失去"用户拥有数据 + 跨工具兼容"

## 后果

### 正面

- 用户在 Finder / VSCode / GitHub 浏览仓库时有清晰导航
- 用户可以为关键文件加备注，备注随文件一起存在
- 改动历史不污染文件系统
- 应用卸载后用户的备注仍然完整保留

### 负面 / 代价

- **README 自动生成需要触发时机**：导入 / 删除 / 重命名后异步重生成
  - 缓解：debounce + 后台 worker
- **标记块机制对用户有学习成本**：用户需要知道"标记块内别改"
  - 缓解：UI 中清晰提示 + 标记块前后加注释解释
- **冲突场景复杂**：用户在外部编辑了标记块内容 → 下次重生成会被覆盖
  - 缓解：重生成前对比，如检测到改动则 fallback 到追加而非覆盖
- **README 文件本身也算 file**：要不要在 files 表里记录？
  - 决策：**不记录**，README 是应用产物不是用户资料

### 风险

- 用户误把 README.md 当作要管理的资料拖到应用里 → 应用要识别并跳过
- 多用户协作（git）下 README 频繁 diff → 提供 `--no-readme` 选项 / 标记块内容确定性排序
- 用户备注的 `<name>.md` 与同名 markdown 文档冲突
  - 缓解：导入 markdown 时检测同目录已有 `<name>.md` 备注 → 重命名为 `<name> (note).md` 或拒绝并提示

## 何时重审

- 用户大量反馈"为什么没有每个文件的 readme" → 重审是否提供 opt-in
- README 重生成成为性能瓶颈（仓库 > 1 万文件）→ 增量生成
- Stage 3 加 AI 后，自动生成"文件摘要" → 决定写到 README 还是单独存
