# 分类器调教（Classifier Calibration）

> 定义用户如何“纠正错误分类”、如何理解“为什么分到这里”、以及如何把纠正动作沉淀为可复用规则（`classifier.yaml`）。本文只规定 UX 与产品策略，不重复实现细节。
>
> 阅读时长：约 18 分钟。

---

## 目标与成功标准

### 目标

1. **纠错成本极低**：用户发现分错 → 2 次点击内完成纠正。\n
2. **规则可沉淀**：每次纠正都可以选择“仅这一次”或“以后都这样”。\n
3. **可解释**：用户能看到“命中了哪些规则/关键词/扩展名”。\n
4. **安全**：任何规则变更都可撤销、可预览影响范围，避免“一改全乱”。\n
5. **兼容两类用户**：新手只点按钮；高级用户可编辑 YAML。\n

### 成功标准（验收）

- **C1**：用户在 Detail 面板点击“分类不对？”→ 选择目标分类 → 立刻生效。\n
- **C2**：用户勾选“记住这个规则”→ 系统在 `classifier.yaml` 写入/更新规则（或给出预览并确认）。\n
- **C3**：“为什么分到这里”能展示：扩展名命中/关键词命中/优先级/兜底。\n
- **C4**：规则变更后提供“预览影响：将影响 N 个文件（可查看）”。\n
- **C5**：YAML 校验失败时给出可定位错误（行号/字段）与回滚按钮。\n

## 入口（用户从哪里开始调教）

### 入口 1：Detail 面板（推荐，最高频）

- 当用户看到文件分类不对，在 Detail 的 Meta 区域旁显示小按钮：`分类不对？`\n
- 这是“就地纠错”，不打断用户心流。\n

### 入口 2：ImportSheet（导入前纠错）

在 `drag-import-flow.md` 的 ImportSheet 中：\n
- 建议分类旁的 `（为什么？）`\n
- 以及分类下拉直接改分类\n
\n
这里的调教目标是“本次导入”与“沉淀规则”二选一。

### 入口 3：设置 → 分类规则

面向高级用户：\n
- `Settings → 分类规则`（见 `settings-panel.md`）\n
- 提供 YAML 编辑、Schema 校验、示例模板、导入导出\n

### 入口 4：命令面板（Cmd+K，Stage 2）

提供命令：\n
- `Calibrate classifier for current file…`\n
- `Open classifier rules…`\n

## 核心交互 1：快速纠错（不沉淀规则）

### UI（Detail → 分类纠错弹窗）

```
┌──────────────────────────────────────────────────────────────────────────────┐
│ 纠正分类：合同_2026Q1.pdf                                                       │
│                                                                              │
│  当前：docs                                                                    │
│  改为： [ finance ▾ ]                                                          │
│                                                                              │
│  说明：这会把文件移动到 finance/ 并更新改动日志。                               │
│                                                                              │
│  [ Cancel ]                                                   [ Apply ]      │
└──────────────────────────────────────────────────────────────────────────────┘
```

### 结果反馈

Apply 后：\n
- List 中该文件从当前分类移除，目标分类计数更新\n
- Detail 显示 banner：`已移动到 finance  [Go to]`\n
- change_log 追加 `Moved`（或 equivalent action）\n

---

## 核心交互 2：纠错并沉淀规则（“以后都这样”）

### UI（在快速纠错弹窗里增加“记住规则”）

```
┌──────────────────────────────────────────────────────────────────────────────┐
│ 纠正分类：合同_2026Q1.pdf                                                       │
│                                                                              │
│  当前：docs                                                                    │
│  改为： [ finance ▾ ]                                                          │
│                                                                              │
│  [✓] 记住这个规则（以后类似文件自动归到 finance）                               │
│      规则依据： (●) 扩展名 .pdf   ( ) 关键词…   ( ) 仅本文件名…                │
│                                                                              │
│  [ Preview impact… ]                                                          │
│                                                                              │
│  [ Cancel ]                                                   [ Apply ]      │
└──────────────────────────────────────────────────────────────────────────────┘
```

### 规则依据（可选项解释）

| 选项 | 含义 | 风险 |
|---|---|---|
| 扩展名 | 对所有 `.pdf` 生效 | 🔴 太宽（不推荐默认） |
| 关键词 | 对含关键词的文件名生效 | 🟡 合理 |
| 仅本文件名 | 精确匹配当前文件名 | 🟢 安全但收益小 |

产品策略建议：\n
- 默认选择 **关键词**（从当前文件名抽取候选关键词）\n
- 扩展名规则必须二次确认（防止“所有 pdf 都进 finance”）\n

## Preview impact（预览影响范围）

### 目的

避免“改一条规则影响一大片”，并给用户可见性。\n

### UI（预览 sheet）

```
┌──────────────────────────────────────────────────────────────────────────────┐
│ 预览规则影响                                                                    │
│                                                                              │
│  规则：关键词包含 “合同” → finance                                             │
│                                                                              │
│  将影响： 37 个文件                                                            │
│                                                                              │
│  [PDF] 合同_2025Q4_客户B.pdf     当前：docs   变更后：finance                   │
│  [PDF] 合同_2026Q1_客户A.pdf     当前：docs   变更后：finance                   │
│  ...                                                                          │
│                                                                              │
│  [ Back ]                                            [ Confirm rule change ] │
└──────────────────────────────────────────────────────────────────────────────┘
```

### 规则生效时机

用户确认后：\n
- 规则写入 `classifier.yaml`\n
- **立即**对受影响文件提供“批量重分类”建议：\n
  - `Apply now (reclassify 37 files)`\n
  - `Later`（仅对未来导入生效）\n

这能同时满足“我想马上整齐”与“我只想以后生效”的两类用户。\n

## “为什么分到这里？”解释面板

### 触发入口

在 ImportSheet 与 Detail Meta 中，分类旁提供 `（为什么？）`。\n

### UI（popover）

```
┌──────────────────────────────────────────────────────────────────────────────┐
│ 为什么归到 docs？                                                               │
│                                                                              │
│  命中：扩展名规则                                                              │
│    • .pdf → docs （priority=80）                                               │
│                                                                              │
│  命中：关键词规则                                                              │
│    • “报告” → docs（priority=60）                                              │
│                                                                              │
│  选择：扩展名优先级更高，因此结果为 docs。                                     │
│                                                                              │
│  [ Change rules… ]                                                            │
└──────────────────────────────────────────────────────────────────────────────┘
```

### 解释字段约定（产品侧）

解释必须包含：\n
- 命中类型（extension/keyword/fallback）\n
- 命中内容（`.pdf` / `合同`）\n
- priority\n
- 决策原因（冲突如何解决）\n

实现细节参见：`docs/modules/classify.md` 与 `docs/api/classifier-yaml.md`。\n

## YAML 编辑模式（高级）

### Settings → 分类规则 页面（概念，细节在 settings-panel.md）

必须提供：\n
- 语法高亮\n
- JSON Schema 校验\n
- 错误提示带行列号\n
- 一键恢复上次有效版本\n

### YAML 保存策略（产品侧）

1. 用户点击 Save → 先校验\n
2. 校验失败 → 不写入生效文件，停留编辑页\n
3. 校验成功 → 写入并提示“规则已更新，对未来导入生效”\n
4. 可选：提供“立即重新分类（N 个文件）”按钮\n

---

## 规则写入策略（产品侧，不涉及实现）

规则写入不应直接把用户的手工 YAML 结构“打乱”。建议：\n

- 如果用户开启“自动维护规则”（默认关）：系统可追加一个 `areamatrix_auto_rules:` 区块\n
- 如果关闭：系统只给出建议并让用户复制粘贴（更尊重高级用户）\n

（具体实现与 marker 类似，可借鉴 readme-gen 的 BEGIN/END 协议，但是否采用由工程决定。）\n

## 附录：默认关键词提取策略（产品侧建议）

当用户选择“关键词”为规则依据时，UI 需要给出候选关键词（可勾选多个）。建议策略：

1. 取文件名（不含扩展名）\n
2. 以分隔符切分：空格、下划线、短横线、中文标点\n
3. 过滤掉纯数字、长度 1 的 token\n
4. 对中文不做分词（先用 contains），给出 2-4 字的连续片段候选（可选）\n

示例：

- `合同_2026Q1_客户A.pdf` → 候选：`合同`、`客户A`、`2026Q1`（数字可选）

（实现细节参见 `docs/modules/classify.md` 的 normalize/tokenize 章节。）
