# 竞品深度对比（Competitive Analysis）

> 对比 Finder / Hazel / DEVONthink / Eagle / Obsidian 在“资料归档 + 自动化 + 可追溯 + 可视化”上的做法，并明确 AreaMatrix 的差异化定位、取舍与不做什么。本文基于公开信息与常见用户口碑，避免臆测竞品内部实现。
>
> 阅读时长：约 18 分钟。

---

## 目标与成功标准

### 目标

1. **明确我们要赢在哪里**：不是“都做”，而是“做对一组核心任务”。\n
2. **明确我们不做什么**：避免做成复杂的 DEVONthink 或纯规则的 Hazel。\n
3. **把差异化落到 UX**：不是一句“更现代”，而是具体交互与默认策略。\n

### 成功标准（验收）

- **CA1**：能用 3 句话解释“为什么用 AreaMatrix 而不是 Finder/Hazel”。\n
- **CA2**：能列出 5 个“我们不追”的能力（避免 scope creep）。\n
- **CA3**：每个差异点都映射到 Stage 1/2 的具体 UX 文档（如导入/调教/可追溯）。\n

---

## 我们的定位（再确认）

> 个人版 DEVONthink 的现代轻量替代 + Hazel 的规则智能 + Obsidian 的活文档思想。\n
\n
更具体：\n
- 以 **资料库（普通文件夹）** 为中心（local-first）\n
- 以 **拖拽导入 + 自动分类/命名** 降低整理成本\n
- 以 **改动时间线 + README** 提供可追溯\n
- 以 **树状导航 + 详情面板** 提供可视化管理\n

---

## 竞品对比维度（8 维）

| 维度 | 我们关心的问题 |
|---|---|
| A. 资料形态 | 数据是否锁定在 app 内？弃用后是否可读？ |
| B. 导入体验 | 归档动作是否“零意志力”？ |
| C. 自动化能力 | 规则/智能/可解释性如何？ |
| D. 搜索能力 | 名称/全文/OCR/语义，体验与成本 |
| E. 可追溯性 | 有没有时间线、版本、操作历史？ |
| F. 可视化管理 | 树/列表/详情，能否批量与多选？ |
| G. 隐私与离线 | 默认是否本地？AI 是否可选？ |
| H. 学习成本 | 上手曲线与复杂度 |

---

## 总览表（高层）

| 产品 | 强项 | 弱项 | 适合谁 |
|---|---|---|---|
| Finder | 系统级、无锁定 | 无自动分类/无历史 | 所有人（基础） |
| Hazel | 规则自动化强 | 无资料库视图、偏后台 | 规则控、自动整理 |
| DEVONthink | 分类/检索极强 | 重、贵、学习成本高 | 专业知识管理重度用户 |
| Eagle | 视觉素材管理 | 文档/代码不友好 | 设计师素材库 |
| Obsidian | 笔记+链接生态 | 二进制资料管理弱 | 笔记/知识网络 |
| AreaMatrix | 拖拽归档 + 可追溯 + 可视化 | 早期搜索弱、需要规则调教 | 个人/小团队资料归档 |

## 逐个竞品分析

### 1) Finder

#### Finder 的“默认路径”

- 以文件系统为中心（与我们一致）\n
- 强在“任何 app 都能打开、任何工具都能操作”\n
- 弱在“自动化与可追溯”\n

#### 我们要补上的

- `drag-import-flow.md`：拖拽导入 → 自动分类/命名\n
- `ui-states.md`：树/列表/详情 + 多选 + 批量\n
- `modules/change-log.md`：时间线\n
- `modules/overview-gen.md`：AreaMatrix 专属概览\n

#### 我们不做的

- 不替代 Finder 的全部能力（标签颜色、智能文件夹生态等），只做“资料库内”的管理。\n

---

### 2) Hazel

#### Hazel 的核心

- 面向“规则驱动的后台自动化”：监控文件夹 → 按规则移动/重命名/执行脚本。\n
- 强在“可编排动作”与“长期无人值守”。\n

#### Hazel 的缺口（对我们来说）

- 缺少“资料库视图”与“可追溯时间线”。\n
- 用户很难回答“我上周改过哪些文件/搬过哪些文件”。\n

#### AreaMatrix 的差异化

- 我们把规则（classifier.yaml）用于“导入时决策”，并提供“纠错→沉淀规则”。\n
- 规则不是目的，是“降低意志力”的手段。\n
- 参见：`classifier-calibration.md`。\n

#### 我们不做的

- 不做 Hazel 那种“任意脚本动作编排”（Stage 4 以后再考虑插件）。\n

---

### 3) DEVONthink

#### DEVONthink 的核心

- 强自动分类、强全文检索（含 OCR）、强组织与引用。\n
- 代价：复杂、学习成本高、UI 偏传统。\n

#### 我们要学的

- 分类的“可解释性”（为什么分到这里）\n
- 资料管理的“可追溯”（时间线、版本）\n

#### 我们要避免的

- 不做“一个系统内的第二套文件系统”。\n
- 不把用户数据锁进专有数据库（我们坚持 repo 是普通文件夹）。\n

#### 差异化落点

- Stage 1：规则透明 + 可调教 + 轻量 UI\n
- Stage 2：搜索（文件名+笔记+元数据）\n
- Stage 3：AI/OCR 作为可选增强（不默认开启）\n

参见：`search.md`、PRD 的 Stage 3。\n

---

### 4) Eagle

#### Eagle 的核心

- 面向“视觉素材”：快速预览、标签、分组、设计师工作流。\n

#### 我们的不同

- 我们是“资料混合”（文档/代码/合同/票据/截图）\n
- 视觉预览不是第一优先（Stage 2 可补 Quick Look）\n

#### 可借鉴点

- 强多选与批量操作\n
- 标签作为横切维度（见 `deep-features.md`）\n

---

### 5) Obsidian

#### Obsidian 的核心

- Markdown 笔记与链接网络，强生态。\n

#### 我们的不同

- 我们以“文件条目”与“资料库结构”作为主对象，而不是笔记。\n
- 我们的 note 是“伴生笔记”（companion note），用于为文件补充说明。\n

#### 可借鉴点

- local-first\n
- 文档化与可链接的知识组织（AreaMatrix 概览 / Note）\n

---

## 结论：我们的 5 个差异化承诺（可写进 README）

1. **Repo 就是普通文件夹**：弃用也能用 Finder 继续管理。\n
2. **拖拽即归档**：导入时自动分类/命名，不需要用户思考。\n
3. **每个动作都有账**：change_log + AreaMatrix 概览。\n
4. **纠错会沉淀**：分类不对？两次点击纠正，并可选择沉淀为规则。\n
5. **本地优先，智能可选**：AI/OCR 不是强制依赖。\n

---

## 我们明确“不做”的清单（防 scope creep）

1. 不做企业级协作与权限系统（Stage 4 以后再说）\n
2. 不做任意脚本自动化编排（Hazel 类能力）\n
3. 不默认开启 AI/OCR（默认完全离线）\n
4. 不做跨云盘多家 SDK 集成（先 iCloud）\n
5. 不做复杂的知识网络（双链/图谱）\n
